## Новые конкретные факты (которых не было в прошлом разборе)

**МойСклад — тарифы с 1 июля 2026** (помесячно / при годовой), [moysklad.ru/subscription](https://www.moysklad.ru/subscription/):
- Базовый — 1400 / 1120 ₽/мес (2 сотр.)
- Проф — 4100 / 3280 ₽/мес (5 сотр.)
- Корп — 9800 / 7840 ₽/мес (10 сотр.)

**МойСклад: маркировка — ПЛАТНАЯ опция, не в базовом тарифе** — «Маркировка» +700 ₽/мес (до 30.06.2026 — 650 ₽). Включает работу с маркированным товаром, интеграцию с Честным знаком, экспорт/импорт УПД, продажи/возвраты через кассу. ⚠ Этого не было в прошлой оценке — закладывать в регулярные расходы.

**МойСклад: партии + FEFO** — учёт по сериям/срокам годности, автоподбор партии по FEFO (раньше истекает — раньше отгружается). [support.moysklad.ru](https://support.moysklad.ru/hc/ru/articles/14133687303569). Под чай (сроки + год сбора) — подходит штатно.

**RetailCRM — цены** [retailcrm.ru/prices](https://www.retailcrm.ru/prices): «Профессиональный» от 9 360 ₽/мес, минимум 3 пользователя, база от 10 000 контактов, +3 200 ₽/мес за доп. пользователя. (Прошлая оценка «4000–5000 ₽/мес» была занижена.)

**RetailCRM — квиз→сегмент технически решается** [docs.retailcrm.ru](https://docs.retailcrm.ru/Users/Administration/SettingBlock/Customfield/CreatingCustomfield): пользовательские поля типа «справочник» хранят результат квиза, и поле можно прямо использовать как условие сегментации. То есть связка «квиз пишет сегмент в CRM» — штатная, не самопис.

**1С-Битрикс — рост цен с 01.01.2026** [aspro.ru](https://aspro.ru/news/povyshenie-tsen-na-1sbitriks-010126/): «Малый бизнес» 47 000 ₽ (было 40 900), «Бизнес» 96 500 ₽ (было 83 900), +~15%.

**InSales — верхний тариф «Ultra»** 29 900 ₽/мес, до 300 000 товаров [insales.ru](https://www.insales.ru/page/obnovlenie-tarifov-insales-v-2026). Для 150–200 SKU достаточно среднего тарифа — Ultra избыточен.

**Фасовочная модель — A + B, перефасовки НЕТ.** Магазин продаёт пачки из коробки как пришло (уже с кодом) + развес на вес. Своих брендированных пачек не делает → эмиссии новых кодов и роли фасовщика НЕТ, GetMark/Клеверенс не нужны. Маркировочный контур лёгкий: приёмка кодов по ЭДО → вывод на кассе (из коробки) и выбытие «Фасовка» по партии (развес); всё закрывает опция «Маркировка» МойСклада.

**Грабли практика (vc.ru)**: связка **Битрикс24 + МойСклад** — каталоги бесшовно НЕ объединяются, нужна ручная дубль-заводка товара/доставки. [vc.ru/187181](https://vc.ru/marketplace/187181). Аргумент против Битрикс24 как CRM в этом стеке.

## Разбор по слоям

### 1. Платформа витрины
- **1С-Битрикс + Аспро** — мощно, коробка, модуль маркировки штатный, но дорожает (лицензия 47–96k + Аспро + разработка). Оправдан при глубокой кастомизации.
- **InSales / AdvantShop** — SaaS, быстрый старт, маркировка и 54-ФЗ из коробки, заметно дешевле. Лучший фит по масштабу (200 SKU, один владелец) — ЕСЛИ не нужен бессшовно кастомный UX.
- **CS-Cart** — коробка, силён на «несколько тысяч SKU» и кастоме, интегрируется с RetailCRM/1С/МойСклад. Для 200 SKU избыточен, но дешевле Битрикса как коробка.
- **WebAsyst Shop-Script** — коробка средней тяжести.
- **headless (Medusa/Saleor + Next/Nuxt)** — максимум гибкости, но маркировку/54-ФЗ/кассы строишь сам. Для 200 SKU и одного владельца не оправдан.

**Вывод по слою:** настоящая развилка — не «Битрикс или ничего», а **Битрикс vs InSales/AdvantShop**. Выбор определяется тем, насколько кастомны квиз и двойная фасовка с брендингом. Если кастом критичен → коробка (Битрикс/CS-Cart). Если важнее скорость и цена → SaaS (InSales/AdvantShop), и тогда бюджет 1.2 млн режется кратно.

### 2. CRM
**RetailCRM** остаётся правильным выбором: e-commerce-нативная, квиз→сегмент через справочные поля штатно, связка с МойСклад документирована. **amoCRM** — под дорогие штучные продажи (не поток), не наш профиль. **Битрикс24** — корпортал «всё-в-одном», но с МойСклад болезненная ручная синхронизация (vc.ru). Цена RetailCRM выше, чем думали (от 9 360 ₽/мес), — поправить смету.

### 3. Склад / маркировка
**МойСклад** закрывает партии + FEFO + маркировку (опция +700 ₽/мес), нативно с RetailCRM — обоснованный выбор под требования. **1С (УТ/Розница)** — мощнее, целевая при росте в цех/опт, дороже на входе. **Контур.Маркет** — силён в маркировке/ЭДО, слабее как ядро. (GetMark/Клеверенс не нужны — сложной фасовки/эмиссии кодов в этом кейсе нет.)

## Итоговая рекомендация (уточнённые вводные)

Вводные: **80–100 SKU**; владелец хочет **настоящий кастомный дизайн и вайбкодить фронт сам**, шаблонный SaaS-вид не устраивает; фасовка **A + B** (из коробки + развес), своих пачек нет.

**Ключевой разворот:** маркировка и 54-ФЗ живут НЕ в витрине, а в бэк-офисе (МойСклад + касса + ЮKassa). Поэтому «кастомный дизайн» и «соответствие закону» не конфликтуют — витрину выбираем по свободе кода, compliance вешаем на бэк-офис.

**InSales вычеркнут из рекомендации.** Проверка (см. ниже) + аггрегаторы примеров показали: витрины InSales шаблонные, кастом упирается в потолок Liquid и «нельзя выйти за пределы функционала». Под требование «настоящий кастом» не подходит.

**Рекомендуемый стек:**
- **Витрина — свой фронт (Next / Astro / SvelteKit), вайбкод.** Полная свобода дизайна, без шаблонного потолка. Заказ уходит по API в RetailCRM.
- **CRM — RetailCRM.** Заказы, клиент, квиз→сегмент штатно.
- **Склад — МойСклад + опция «Маркировка» (+700 ₽/мес).** Партии, сроки/FEFO, коды ЧЗ, выбытие.
- **Платежи — ЮKassa** (оплата + фискальный чек 54-ФЗ).
- Цена свободы: сам пишешь склейку чекаут → оплата → фискализация → выбытие кода. Это реальная работа (валидировать при реализации).

**Средний путь (кастом без склейки с нуля): CS-Cart** — коробка на своём сервере с полным доступом к коду и шаблонам (не Liquid-песочница), интеграции с RetailCRM/МойСклад, чекаут/платежи/маркировка из коробки.

**Не Битрикс+Аспро за 1.2 млн:** для 80–100 SKU тяжёлая коробка избыточна; основная часть 1.2 млн — труд разработчиков, который владелец берёт на себя вайбкодингом.

**Регулярка:** «Маркировка» МойСклада (+700 ₽/мес) + RetailCRM (от 9 360 ₽/мес) + хостинг своего фронта. Кратно дешевле разработки за 1.2 млн.

## InSales — проверка реальной кодабельности (VERIFIED по докам + отзывам, 17.06.2026)

Два РАЗНЫХ вопроса, их нельзя смешивать:

**A. Кодить тему (вёрстка фронта) — ДА.** Темы = Liquid + сниппеты (HTML+Liquid). Редактор кода 4-го поколения правит HTML/SCSS/JS/JSON + Liquid, с функциями VS Code (подсветка, Emmet, автодополнение). Есть фронтовый JS/AJAX API (корзина, каталог, фильтры). Оговорки: официальный редактор — веб внутри админки; выгрузка темы/git/локальная разработка официально НЕ документированы (есть только «История изменений» для отката); локальный тулинг — community (uploader, GitHub VladimirIvanin/insales-theme); сборка CSS/JS — внутренними директивами платформы, не свой webpack.

**B. Подключение по REST API — ДА, но это АДМИН-API, не storefront.** REST, Basic Auth (`password=MD5(token+secret_key)`). Назначение (дословно): «создавать, читать и удалять объекты панели администратора» + вебхуки. Лимит 500 запросов/5 мин на магазин, 503 при превышении. Библиотеки: insales/insales_api (Ruby), npm insales/node-insales, pyinsales.

**Headless-вывод:** свой Next/React фронт на InSales технически возможен, но через админский data-API (Basic Auth с секретом — нельзя в браузер), с лимитом 500/5мин, без customer-facing storefront-API. Это самодельщина через свой бэкенд-прокси, НЕ родной headless. Если нужен первоклассный headless — Medusa/Saleor + свой фронт, либо CS-Cart (полный код на своём сервере).

Цитаты-минусы (otzyvmarketing.ru/insales): «Нельзя выйти за пределы функционала» (Г. Верещагина); «Шаблонов катастрофически мало» (Е. Никипелов); «Низкая скорость сайтов, недоступны по 1-2 часа» (Р. Хисматуллин); поддержка — «за 12 месяцев ни один вопрос не решён» (Артем А).

Развилка для финала: «вайбкодить дизайн» = (а) правка темы InSales → InSales подходит; (б) свой отдельный фронт на React/Next → InSales не та платформа, смотреть headless (Medusa/Saleor) или CS-Cart.

Источники: insales.ru (редактор кода 4-го поколения, дока API/интеграции), github.com/liquid-hub/insales-common-js-v2-api, github.com/VladimirIvanin/insales-theme, github.com/insales/insales_api, otzyvmarketing.ru/insales.

# Финальный стек — интернет-магазин китайского чая

Решение после совместного разбора. Вводные: ~80–100 SKU; кастом-дизайн важен; разработка бесконечна ДО запуска, после — ноль; приоритет «работает и не падает»; фасовка A+B; маркетинг сложный — потом.

## Стек

| Слой | Решение |
|---|---|
| **Витрина** | **Tilda.** SaaS — вендор держит, ноль обслуживания. Дизайн кодишь сам в Zero Block (HTML/CSS/JS), хостит Тильда |
| **Квиз** | Родной квиз Tilda: ветвление → шаг «Результат: редирект» → отфильтрованный каталог или страница-подборка. Без кнопок, без CRM. Сложная логика — вайбкод JS в Zero Block |
| **Карточка товара** | Родной store-блок + свой CSS (ноль зависимостей). Радикальный кастом карточки — модификация Nolim NLM151 (платно, +зависимость) — только если реально нужно |
| **Склад** | **МойСклад + опция «Маркировка»** (+~700 ₽/мес). Партии, сроки/FEFO, остатки, резерв |
| **Оплата** | **ЮKassa** — приём + фискальный чек 54-ФЗ |
| **CRM** | **НЕТ на старте.** Квиз→редирект CRM не требует. RetailCRM добавляется слоем позже |
| **Данные/трекинг** | Квиз → cookie → скрытое поле заказа → сегмент в МойСкладе; Google Sheets для не-купивших; Яндекс.Метрика для метрик по сегментам |

Связки (готовые, не кодишь): Tilda↔МойСклad (родная синхронизация товаров/остатков), Tilda→заказ→МойСклад (коннектор, доп. поле для сегмента).

## Фасовка и маркировка

- Модель **A** (продажа пачек из коробки как пришло, уже с кодом) **+ B** (развес на вес). Своих брендированных пачек НЕТ → нет эмиссии кодов, нет роли фасовщика, GetMark не нужен.
- Маркировка живёт **только на складе**: приёмка кодов по ЭДО + выбытие «Фасовка». Витрина и касса маркировку не трогают.
- Операция «Фасовка» в МойСкладе делает сразу две вещи: переводит кг-кучу в счётные навески (для витрины) и снимает код ЧЗ.
- Весовой чай на витрине = товар с вариантами-навесками (50/100/250 г), счётные позиции. Не «цена за грамм».

## Закон (152-ФЗ)

Данные покупателей на русском профиле Тильды хранятся в РФ (Selectel/Яндекс Облако). Нужно: регистрация оператором ПД в РКН + политика конфиденциальности + чекбоксы согласия (инструменты у Тильды есть). Обязанность оператора — на любом стеке, не минус Тильды.

## Принципы (почему так)

- **«Ноль поддержки после запуска» → SaaS.** Тильда/МойСклад/ЮKassa держат вендоры. Коробка (Битрикс/CS-Cart) и самопис ОТВЕРГНУТЫ — гниют без обслуживания.
- **Свой код = только клиентский вайбкод внутри Тильды** (Zero Block). Хостит Тильда, своего сервера нет → рот-риск низкий.
- **InSales отвергнут** (шаблонные витрины). Headless/самопис отвергнут (гниёт при нуле поддержки).

## Future-proof

Коммерческая история (заказы, клиенты) копится в МойСкладе — **открытый JSON API**, данные портативны. RetailCRM добавляется позже: подключается к МойСкладу (родная связка) + Тильде (коннектор), история импортируется. Магазин не пересобирается.

Что НЕ восстановить задним числом — то, что не сохранял (квиз-сегмент без записи, поведение). Поэтому сегмент пишем в заказ уже сейчас (страховка). Tilda «Заявки» хранятся 30–90 дней — долговременный архив только МойСклад.

## Поток данных

```
Покупатель → квиз (Tilda) ─┬─→ редирект на подборку (отфильтрованный каталог)
                            └─→ сегмент в cookie
   → корзина (Tilda, товары из МойСклада)
   → оформление: cookie → скрытое поле quiz_segment
   → ЗАКАЗ
       ├─→ ЮKassa (оплата + чек 54-ФЗ)
       ├─→ МойСклад (заказ + сегмент; история; выбытие «Фасовка»)
       ├─→ Google Sheets (сегмент не-купивших)
       └─→ Яндекс.Метрика (метрики/воронка по сегментам)
[позже] RetailCRM ← импорт истории ← МойСклад (открытый API)
```

## Что ещё уточнить при внедрении

- Тариф коннектора Tilda↔МойСклад для передачи доп. поля (сегмента) в заказ.
- Глубину вёрстки store-блока карточки (если нужен радикальный кастом — Nolim).
- Актуальный список дата-центров Тильды для уведомления в РКН.
